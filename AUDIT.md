# SwarmVault — Codebase Audit

> Comparison of README.md documentation against the actual implementation.
> Generated after a full read of all server routes, services, desktop sync code, and Prisma schema.

---

## Section 1 — In README / docs but NOT (fully) implemented in code

| #   | Item                                               | Details                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| --- | -------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **`PUT /api/v1/files/:id/share`**                  | Listed in the README files table as a dedicated update endpoint. `sharing.ts` has no `PUT` route — only `POST /:id/share`, which handles create/update internally via an upsert. The desktop `sync.ts` also calls `POST` for both create and update. The `PUT` endpoint does not exist.                                                                                                                                                      |
| 2   | **Swarm Claim Flow — desktop UI trigger**          | README step 1 says "User marks file as claimed in the desktop app." There is no claim button in `FileManager.tsx`. The `POST /api/v1/files/:id/claim` server endpoint works, but the UI never calls it. The "claimed" status icon renders correctly, but there is no way for the user to initiate the claim flow from within the app.                                                                                                        |
| 3   | **Production shard config (4+2)**                  | README mentions "4 data + 2 parity shards in production; 2+1 during beta" throughout. No production-profile switch exists in code — `DEFAULT_DATA_SHARDS = 2` and `DEFAULT_PARITY_SHARDS = 1` are the only values in `constants.ts`, with a comment to "restore to 4+2 for production".                                                                                                                                                      |
| 4   | **Reward tier multipliers in Storage Tiers table** | The "Storage Tiers" table in README says Vault = **1.5×** and Swarm = **1.0×**. The actual `TIER_MULTIPLIER` constant (and the Reward System table in the same README) correctly shows `vault: 0.5` and `swarm: 0.2`. The two README sections directly contradict each other.                                                                                                                                                                |
| 5   | **Reward formula documentation**                   | README shows `credits_per_hour = (avg_pledged_gb × uptime_pct × tier_multiplier) / 24`. The real `rewardService.runSnapshot()` applies a 3-month rolling average uptime (`uptimePct3m`) when ≥ 24 snapshots exist, plus an undocumented **uptime ramp**: nodes below 50% uptime earn **zero credits**, 50–80% earns a linear fraction, ≥ 80% earns full credit. Neither the ramp nor the 3-month averaging logic is mentioned in the README. |
| 6   | **`GET /api/v1/files/trash`**                      | There is a `GET /trash` route in `files.ts` that lists trashed files. It is not present in the README files API table. The desktop `FileManager.tsx` uses it to render the Trash tab, so it is clearly production-important.                                                                                                                                                                                                                 |

## Section 2 — Implemented in code but NOT documented in README

### Server — REST endpoints missing from API Reference

| Endpoint                                 | Notes                                                                                                                                                                                                    |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `PUT /api/v1/files/:id`                  | Re-uploads a file in place. Preserves the file ID (so active shares survive), wipes old chunks, resets to `pending`, and triggers a new shard assignment. Used by `sync.ts` when a watched file changes. |
| `GET /api/v1/files/:id`                  | Returns full file record including chunks and their node locations.                                                                                                                                      |
| `GET /api/v1/files/manifest`             | Lightweight reconcile feed: returns `id, path, contentHash, status, updatedAt` for every non-deleted file. Used by `sync.ts` at startup (`refreshManifest`).                                             |
| `GET /api/v1/files/:id/assignment`       | Polling endpoint for queued uploads. After a `202` response from `POST /files`, the desktop polls here until a shard assignment is available. Checks Redis cache, falls back to live node selection.     |
| `GET /api/v1/files/:id/retrieval-status` | Returns the latest `RetrievalJob` for a file. Useful for polling Swarm Claim job progress. Not exposed in the desktop IPC layer either.                                                                  |
| `GET /api/v1/files/:id/share`            | Returns the active share token, URL, and expiry for a file (owner only). Used internally by `sync.ts` in `refreshShareShadow()` to check whether a re-upload should refresh the shadow copy.             |
| `DELETE /api/v1/nodes/:id`               | Unregisters a storage node (hard delete). Implemented in `nodes.ts`, not in the README nodes table or the desktop Settings UI.                                                                           |
| `GET /api/v1/rewards/leaderboard`        | Public endpoint returning the top 25 contributors by lifetime earned credits. Not in the README rewards table.                                                                                           |
| `GET /health`                            | Returns `{ status: "ok", ts: <ISO> }`. Standard health-check endpoint, not documented.                                                                                                                   |

### WebSocket — server→node messages missing from WS section

| Message type           | Sent when                             | Notes                                                                                                                |
| ---------------------- | ------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| `auth_ack`             | Node WS auth succeeds                 | `{ type: "auth_ack", payload: { nodeId } }`                                                                          |
| `heartbeat_ack`        | WS heartbeat processed                | `{ type: "heartbeat_ack", payload: { ts: <epoch ms> } }`                                                             |
| `chunk_delete`         | Chunk redistributed off a node        | `{ type: "chunk_delete", payload: { fileId, shardIndex } }` — tells the node to delete its local copy                |
| `retrieval_job_update` | Retrieval job status changes          | `{ type: "retrieval_job_update", payload: { jobId, status } }` — triggers the desktop to download the assembled file |
| `error`                | Auth fails or unauthenticated message | `{ type: "error", payload: { message } }` — WS closes with code 4001 on bad credentials                              |

### Server — background behaviours not documented

| Behaviour                                  | Details                                                                                                                                                                                                                                                                                                                                                    |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Automatic chunk redistribution**         | When a node heartbeats with a reduced `pledgedBytes` that is below its `usedBytes`, `redistributeOverCapacityChunks()` is called. It requests chunk data from the over-capacity node (`chunk_request`) and relays it to the least-loaded alternative node (`chunk_relay`), then updates `ChunkLocation` in the DB and tells the source to delete the copy. |
| **Tier auto-promotion/demotion**           | A BullMQ maintenance job runs hourly. Nodes with `uptimePct ≥ 80` are promoted to `vault`; nodes with `uptimePct < 80` are demoted to `swarm`. Initial `tier` at registration is still client-supplied but will be corrected on the next promotion run.                                                                                                    |
| **Trash auto-purge**                       | Soft-deleted files are hard-deleted after 30 days via a BullMQ maintenance job. Not mentioned in the README DELETE endpoint description.                                                                                                                                                                                                                   |
| **Share shadow auto-cleanup**              | Expired `SharedFile` records are purged hourly by a maintenance worker.                                                                                                                                                                                                                                                                                    |
| **Pending distribution retry**             | Every minute, the `node-health` BullMQ worker calls `tryDistributePendingFiles` to retry shard assignment for files that were queued when not enough nodes were online. Also triggered immediately when a node transitions `offline → online` via HTTP heartbeat.                                                                                          |
| **402 Payment Required on quota exceeded** | `POST /files` and `PUT /files/:id` return HTTP 402 (not 413) when the upload would exceed the user's quota. Not mentioned in the README.                                                                                                                                                                                                                   |

### Desktop client — IPC / features not documented

| Feature                                              | Details                                                                                                                                                                                                                       |
| ---------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Assignment polling**                               | When `POST /files` returns 202 (queued), `sync.ts` polls `GET /files/:id/assignment` with exponential backoff. Not mentioned in client-facing docs.                                                                           |
| **Selective sync (`SyncSelector`)**                  | Users can toggle per-folder sync on/off via the Settings page. Deselecting a folder removes local files without trashing them on the server. Not documented in README.                                                        |
| **File conflict handling**                           | During `reconcile()`, if both the local and server copies exist with different hashes and the server version is newer, the local copy is renamed to `<name> (conflict YYYY-MM-DD)<ext>` and the server version is downloaded. |
| **Share shadow refresh on update**                   | When a file that has an active share is re-uploaded, `sync.ts` automatically re-POSTs the new plaintext to `POST /files/:id/share` to keep the shadow copy fresh.                                                             |
| **`X-Is-Data` chunk header**                         | The `POST /api/v1/chunks` request sends an `X-Is-Data: true/false` header distinguishing data shards from parity shards. Omitted from the README chunks header list.                                                          |
| **100 MB file size limit**                           | `sync.ts` enforces a hard 100 MB per-file limit client-side (`MAX_FILE_BYTES = 100 * 1024 * 1024`). Not documented anywhere.                                                                                                  |
| **`contribution:pause` / `contribution:resume` IPC** | IPC handlers exist to pause/resume the storage contribution (sets the WS heartbeat status to `maintenance` / `online`). No corresponding UI element in the renderer.                                                          |
| **`files:downloadToSync` IPC**                       | Allows the renderer to request an on-demand download of a vault file into the sync folder. Not documented.                                                                                                                    |

## Section 3 — Bugs and inconsistencies

| #   | Severity   | Item                                                        | Details                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --- | ---------- | ----------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **High**   | **Redistribution sends base64 JSON `chunk_relay`**          | `redistributeOverCapacityChunks()` in `server/src/index.ts` sends `chunk_relay` as a JSON text frame with `data: chunkData.toString("base64")`. The desktop client (`sync.ts`) now only handles `chunk_relay` as a **binary frame** (4-byte length prefix + meta JSON + raw bytes). When redistribution fires, the target storage node's desktop client silently ignores the JSON-format relay message and never acks, causing the redistribution to time out. The redistribution relay needs to use the same binary format as the regular upload relay. |
| 2   | **Medium** | **`PUT /api/v1/files/:id/share` in README does not exist**  | `sharing.ts` never registers a `PUT` route. The `POST /:id/share` upsert covers both create and update, but any client following the README strictly and calling `PUT` will receive a 404. Either add the `PUT` route as an alias or remove it from the README.                                                                                                                                                                                                                                                                                          |
| 3   | **Medium** | **README Storage Tiers table wrong multipliers**            | "Storage Tiers" section says Vault = 1.5×, Swarm = 1.0×. Actual constants and "Reward System" table both say 0.5 / 0.2. The tier table entry is wrong and should match the constants.                                                                                                                                                                                                                                                                                                                                                                    |
| 4   | **Low**    | **Node `tier` is still client-chosen at registration**      | `POST /api/v1/nodes` accepts and stores the client-supplied `tier`. Comments in `files.ts` say "tier is no longer client-chosen — it is auto-assigned", but that refers to _file_ tier, not node tier. Node tier auto-correction only happens on the next hourly promotion run. README and code comments should clarify that node tier is seeded from the client but corrected by the server.                                                                                                                                                            |
| 5   | **Low**    | **WS heartbeat is duplicated**                              | The desktop can send heartbeats via WS (`sync.ts` `handleWsMessage`) **or** via HTTP `POST /api/v1/nodes/heartbeat`. The WS path in `index.ts` is marked "legacy — HTTP POST preferred" but both paths remain active. The README key data flows table only shows the HTTP path. If the WS path is truly deprecated it should be removed; if it is still used it should be documented.                                                                                                                                                                    |
| 6   | **Low**    | **`uptimePct` does not accumulate correctly for new nodes** | The reward service falls back to `uptimePct` (7-day window, default 0) when fewer than 24 hourly snapshots exist. A brand-new node starts with `uptimePct = 0` in the DB — it only gets updated by the `node-health` worker which calls `nodeService.markStaleNodesOffline()`. If the node is online but the uptime calculation isn't run before the first reward snapshot, the node may incorrectly receive 0 credits on day 1.                                                                                                                         |
| 7   | **Low**    | **`FileStatus.degraded` is unused in client logic**         | The `degraded` status is defined in the Prisma enum and rendered with an amber indicator in `FileManager.tsx`, but nothing in the server code ever sets a file to `degraded`. There is no integrity-check worker that would downgrade `available` → `degraded` when a node goes offline and drops below the minimum shard count.                                                                                                                                                                                                                         |

## Section 4 — Features in neither README nor code (planned / missing)

These are capabilities a production-grade distributed storage system would typically have but are currently absent in both docs and implementation:

- Password reset / forgot-password email flow
- Email verification on registration
- Account deletion
- File rename / move (only full re-upload via `PUT /files/:id` is available)
- File versioning / history
- Desktop auto-update (Electron `autoUpdater`)
- Node integrity verification (the server never re-requests a shard to verify it is still intact after initial upload — `ChunkLocation.verified` is set to `true` on upload but never re-checked)
- Rate limiting on chunk upload endpoint (`POST /api/v1/chunks` is not rate-limited, only global 200 req/min applies)
- Admin / operator API (no way to inspect the system state beyond `swarm-stats` and the leaderboard without direct DB access)
